# Decision Ledger

| Step | Decision | Reason | Rejected options and why | Date |
|------|----------|--------|--------------------------|------|
| 1 | Treat the current repo as a lightweight web workload for sizing. | The app is a single Next.js service with light reads/writes and no heavy compute jobs. | Size it as if it already needed distributed infra: rejected because the current product shape does not justify it. | 2026-04-13 |
| 2 | Set `2 vCPU / 4 GB RAM` as the first-server floor. | This is the smallest spec likely to keep a Node 22 + Next.js runtime comfortable for demos and small pilot traffic. | `1 GB` and `2 GB` memory tiers: rejected because they leave too little runtime headroom. | 2026-04-13 |
| 3 | Keep the first deploy single-node. | The current app still uses local-file-backed demo persistence, so simple stateful deployment is the correct near-term fit. | Multi-node architecture now: rejected because the operational complexity is premature. | 2026-04-13 |
| 4 | Use light-server products only for demo / small pilot and plan to move to ECS/CVM when network guarantees or concurrency needs rise. | Official provider docs position the light-server products for lighter workloads and explicitly steer heavier scenarios toward ECS/CVM. | Assume the light-server class is the permanent production architecture: rejected because it weakens the long-term deployment path. | 2026-04-13 |
| 5 | Compare vendors by stage fit rather than pretending one global “best provider” exists. | The right answer changes materially between offshore demo, mainland pilot, and more formal production stages. | Choose a provider only on headline price: rejected because region, compliance, and service maturity matter more than raw instance cost. | 2026-04-13 |
| 6 | When migration and maintenance cost are included, treat `1 app server + managed PostgreSQL + object storage` as the preferred medium-term baseline. | It raises monthly cost moderately but lowers hidden team cost, data risk, and later rework enough to win on total cost of ownership. | Staying on all-in-one single-node longer: rejected because it pushes state, backup, and migration pain into a later, riskier phase. | 2026-04-14 |
| 7 | Publish a separate deployment-checklist and budget-breakdown artifact instead of overloading the comparison report. | The comparison report answers strategy; the next question was operational: what exactly to buy and what budget band to reserve. | Keep all details buried in the strategy report: rejected because it would make execution slower and harder to reference. | 2026-04-14 |
| 8 | Keep `腾讯云 Lighthouse 2核4GB` as the default short-term domestic demo bundle. | Tencent Cloud’s public mainland Lighthouse pricing is straightforward and unusually cost-effective for a first demo server. | Make B-Lite the default even for a 4-8 week demo: rejected because it spends more money and setup time than the demo stage needs. | 2026-04-14 |
| 9 | For medium-term maintainable pilot budgeting, prefer `阿里云 B-Lite` domestically and `DigitalOcean B-Lite` overseas. | Their public app + managed DB + object storage pricing is easier to explain and reserve budget against than Tencent Cloud’s formula-heavy PostgreSQL pricing. | Use Tencent Cloud as the universal B-Lite recommendation: rejected because the database floor price is harder to defend with a simple public quote. | 2026-04-14 |

## Notes

- Region choice is still conditional on whether the next launch is offshore internal testing or mainland public-facing pilot.
- Pricing confidence is highest where vendors publish fixed bundle prices directly; Tencent Cloud PostgreSQL remains partially inference-based because the public pricing path is formula-oriented rather than bundle-priced.
