# Smart Training Workbench 服务器选型与云厂商对比报告

- 日期：2026-04-13
- 适用阶段：当前 Web-first、mobile-first 的教练端 / 学员端产品阶段
- 结论摘要：
  - 近期目标如果是内部 demo 或小范围封闭试用，优先选择 `单机轻量架构`。
  - 第一台服务器建议从 `2 vCPU / 4 GB RAM / 60-90 GB SSD` 起步，不建议买 `1 GB` 或 `2 GB` 内存机器作为主环境。
  - 如果接下来是中国内地真实 pilot，优先国内云；如果先做内部验证或海外节点试用，可先上香港 / 新加坡。

## 1. 我们的需求

## 1.1 当前产品形态

- 当前代码库是一个 `Next.js + React + TypeScript` Web 应用。
- 目前只有一个可部署应用，承载：
  - 教练端 mobile-first Web
  - 学员端 mobile-first Web
- 当前工作负载主要是：
  - 页面读取
  - 轻量表单提交
  - 基础 API 读写
- 当前不包含：
  - 视频转码
  - 大模型推理
  - 大规模异步任务
  - 搜索集群
  - 高吞吐实时消息系统

## 1.2 从业务角度看，我们真正需要的不是“大机器”，而是“稳、易部署、可升级”

- 近期重点不是超高并发，而是把产品跑通。
- 第一阶段要解决的是：
  - 能稳定部署
  - 能被教练和学员访问
  - 有基础 HTTPS、备份、日志和监控
  - 后续可以平滑升级到数据库和更正式的生产架构
- 也就是说，我们当前的需求本质上是：
  - `轻量 Web 应用托管`
  - `低到中等并发`
  - `中国用户访问延迟可控`
  - `后续可扩容`
  - `运维复杂度低`

## 1.3 容量需求判断

- 按当前产品形态判断，第一阶段更像：
  - 小范围内部 demo
  - 早期产品验证
  - 封闭试用
- 对服务器的真实要求主要体现在：
  - Node.js / Next.js 的运行内存不能太紧
  - 需要有足够磁盘留给日志、构建产物、备份和后续数据库迁移
  - 需要稳定的公网访问和 SSL 配置能力
- 工程判断：
  - `2 GB RAM` 理论上可能跑得动，但留给进程、反向代理、日志和系统自身的余量太小
  - `4 GB RAM` 才是第一台主环境服务器更合理的下限

## 1.4 合规与地域需求

- 如果域名最终解析到中国内地服务器并对外提供 Web 服务，需要走 ICP 相关流程。
- 如果先放在中国香港 / 新加坡等非中国内地节点，可以先跳过 ICP，但中国内地访问延迟会更高。
- 因此我们不是单纯在买一台服务器，而是在做：
  - `阶段性验证环境`
  - `中国内地正式上线环境`
  之间的取舍。

## 2. 架构方案

## 方案 A：单机轻量架构

### 结构

- 1 台应用服务器
- Nginx / Caddy 反向代理
- 1 个 Next.js 应用进程
- 本地磁盘存放日志和临时文件
- 备份先以快照和文件级备份为主

### 推荐规格

- `2 vCPU / 4 GB RAM / 60-90 GB SSD`

### 适用场景

- 内部 demo
- 设计评审
- 小范围封闭试用
- 还没进入真实生产数据阶段

### 为什么这样设计

- 当前应用负载轻，不值得一开始就拆成多服务。
- 单机架构最容易部署、最低成本、问题定位最快。
- 当前系统最大的风险不是“机器不够大”，而是“架构太早变复杂，团队被运维拖慢”。

### 缺点

- 单点故障明显。
- 不适合承载真实生产数据。
- 扩容能力有限。
- 当前如果继续沿用本地文件存储，就不适合多实例横向扩容。

### 结论

- 这是当前最适合的起步方案。

## 方案 B：应用服务器 + 托管数据库

### 结构

- 1 台应用服务器
- 1 个托管 PostgreSQL
- 1 个对象存储
- 反向代理 + HTTPS
- 基础监控和自动备份

### 推荐规格

- 应用服务器：`4 vCPU / 8 GB RAM / 80-120 GB SSD`
- 数据库：托管 PostgreSQL 入门规格

### 适用场景

- 第一批真实试用用户
- 学员反馈、计划版本、操作审计都要持久化
- 需要明显降低“数据丢失”风险

### 为什么这样设计

- 一旦开始接真实教练和学员数据，最大的升级点不是 CPU，而是数据库与可恢复性。
- 托管数据库比自建数据库更适合当前阶段：
  - 少掉数据库运维成本
  - 自动备份更容易做
  - 故障恢复更规范
- 对象存储应提前纳入架构，因为后续极可能会有：
  - 训练附件
  - 截图
  - 导出文件
  - 媒体素材

### 缺点

- 成本比单机高。
- 运维对象从 1 个变成 3 类：
  - 应用服务器
  - 数据库
  - 对象存储

### 结论

- 如果你判断接下来不是“看 demo”，而是“让真实用户开始用”，那这才是更稳的最小生产方案。

## 方案 C：双应用节点 + 负载均衡 + 高可用数据库

### 结构

- 2 台应用服务器
- 1 个负载均衡
- 1 个高可用托管 PostgreSQL
- 1 个对象存储
- 基础告警、监控、自动备份
- 可选 Redis 缓存

### 推荐规格

- 应用节点：`2 x 4 vCPU / 8 GB RAM`

### 适用场景

- 面向外部公开 beta
- 需要更高可用性
- 希望做到无感滚动发布或灰度发布

### 为什么这样设计

- 双节点的核心价值不是“跑得更快”，而是“某一台挂了也还能服务”。
- 负载均衡让发布和扩容更平滑。
- 高可用数据库把最关键的数据风险降下来。

### 缺点

- 成本显著增加。
- 运维复杂度比前两种方案高一个层级。
- 当前阶段如果直接上这套，会过度设计。

### 结论

- 这是“公开 beta / 早期生产”的方案，不是当前第一步。

## 3. 厂商对比与建议

- 说明：
  - 以下价格锚点以 `2026-04-13` 可见的官网公开信息为准。
  - 不同地域、税率、促销、带宽和套餐组合会带来差异。
  - 我这里重点比较的是“和我们当前阶段最相关的计算实例价格与配套能力”。

## 3.1 国内云厂商

## 腾讯云

### 适合我们的原因

- 国内访问体验和网络路径更适合中国用户。
- `Lighthouse` 非常适合当前这种轻量 Web 应用。
- 后续如果升级，可以平滑切到：
  - `CVM`
  - `TencentDB`
  - `COS`
  - `CLB`

### 官网公开价格锚点

- `Lighthouse 2 vCPU / 2 GB / 40 GB SSD / 20 Mbps / 0.5 TB`：`$4.2/月`
- `Lighthouse 2 vCPU / 4 GB / 90 GB SSD / 30 Mbps / 2.5 TB`：`$8.5/月`

### 配套服务情况

- 轻量服务器：有
- 云服务器：有
- 托管数据库：有
- 对象存储：有
- 负载均衡：有
- 监控与基础安全：有

### 如果部署方案 A

- 很合适。
- 可以直接从 Lighthouse 起步。

### 如果部署方案 B

- 合适。
- 一般做法是：
  - 应用层继续用 Lighthouse 或迁到 CVM
  - 数据库上 TencentDB
  - 文件上 COS

### 如果部署方案 C

- 适合。
- 会更自然地进入腾讯云标准云产品体系，而不是继续停留在轻量实例。

### 总结

- 如果我们要优先服务中国用户，又想先低成本上线，腾讯云是非常强的候选。

## 阿里云

### 适合我们的原因

- 国内生态成熟，文档、备案、后续升级路径都完整。
- `轻量应用服务器` 很贴合网站建设、开发测试和小型 Web 应用。
- 后续升级路径清晰：
  - `ECS`
  - `RDS`
  - `OSS`
  - `SLB`

### 官网可见价格情况

- 阿里云官方帮助文档明确说明轻量应用服务器的套餐价格与地域相关，具体价格以控制台购买页为准。
- 官方产品页当前可见的是促销入口和活动信息，例如“低至 68 元/年”的活动展示，但这类活动价不适合作为正式容量预算基线。

### 配套服务情况

- 轻量服务器：有
- 云服务器：有
- 托管数据库：有
- 对象存储：有
- 负载均衡：有
- ICP 路径与帮助文档：很清晰

### 如果部署方案 A

- 合适。
- 但正式预算要以控制台实际地域价格为准。

### 如果部署方案 B

- 很合适。
- 阿里云这条升级路径非常标准：
  - SWAS / ECS + RDS + OSS

### 如果部署方案 C

- 也合适。
- 如果未来明确面向中国内地公开访问，阿里云会更适合作为“正式生产路线”的候选之一。

### 总结

- 如果我们更重视国内正式上线、合规链路和长期生产路径，阿里云是非常稳的方案。

## 3.2 国际云厂商

## AWS Lightsail

### 适合我们的原因

- 上手简单，产品体系完整。
- 适合把轻量 Web 应用快速部署起来。
- 后续也容易迁移到 AWS 更完整的服务体系。

### 官网公开价格锚点

- `2 vCPU / 2 GB / 60 GB SSD`：`$12/月`
- `2 vCPU / 4 GB / 80 GB SSD`：`$24/月`
- Lightsail 负载均衡：`$18/月`

### 配套服务情况

- 轻量实例：有
- 托管数据库：有
- 对象存储：有
- 负载均衡：有
- 全球区域：有

### 如果部署方案 A

- 能做，而且很省心。
- 但对当前项目来说，性价比不如国内轻量云或 Hetzner。

### 如果部署方案 B

- 合适。
- 更偏“标准国际 SaaS 起步方式”。

### 如果部署方案 C

- 合适，但成本会迅速拉高。

### 总结

- AWS 更像“稳妥但偏贵”的国际路线，适合以后要走更重的全球化基础设施时使用。

## DigitalOcean

### 适合我们的原因

- 产品清晰、开发者友好、部署心智负担低。
- 很适合作为海外 demo、原型和小团队产品环境。

### 官网公开价格锚点

- `2 vCPU / 4 GB / 80 GB SSD`：`$24/月`
- `4 vCPU / 8 GB / 160 GB SSD`：`$48/月`

### 配套服务情况

- Droplets：有
- 托管数据库：有
- 对象存储：有
- 负载均衡：有

### 如果部署方案 A

- 合适。
- 海外验证环境会很好用。

### 如果部署方案 B

- 也合适。
- 比 AWS 更轻巧，但生态宽度稍窄。

### 如果部署方案 C

- 可以做，但如果要更复杂的全球架构，AWS 通常更稳。

### 总结

- 如果我们先做海外验证、重视简单和开发效率，DigitalOcean 是好选项。

## Hetzner Cloud

### 适合我们的原因

- 价格性能比极强。
- 很适合预算敏感、偏自托管、偏工程师友好的团队。

### 官网公开价格锚点

- 成本优化型大约从 `2 vCPU / 4 GB / 40 GB` 的 `€3.49-€4.09/月` 起
- `4 vCPU / 8 GB / 80 GB` 大约在 `€5.49-€6.59/月`

### 配套服务情况

- 云服务器：有
- 负载均衡：有
- 对象存储：有
- 私有网络、防火墙、API：有

### 如果部署方案 A

- 很合适，尤其适合内部环境、测试环境、成本敏感环境。

### 如果部署方案 B

- 可以做，但需要更仔细规划数据库和运维策略。

### 如果部署方案 C

- 不是不能做，但更适合强自托管能力的团队。

### 总结

- Hetzner 适合“预算优先、全球不是第一要求、团队接受更强自管”的路线。
- 如果目标用户主要在中国内地，它不是首选。

## 4. 综合建议

## 4.1 如果我们现在就要做决策

- 我会把建议拆成两个版本：

### 建议 A：先做内部验证 / demo

- 地域：`香港` 或 `新加坡`
- 架构：`方案 A`
- 规格：`2 vCPU / 4 GB / 60-90 GB SSD`
- 厂商优先级：
  1. `腾讯云 Lighthouse`
  2. `DigitalOcean`
  3. `Hetzner`

### 为什么这样选

- 当前产品还在快速迭代，不该让备案和重运维拖慢节奏。
- 这时最重要的是：
  - 快速上线
  - 稳定可访问
  - 成本低
  - 能随时迁移

## 4.2 如果我们下一步就是中国内地真实 pilot

- 地域：`中国内地`
- 架构：`方案 B`
- 规格：
  - 应用层先从 `2 vCPU / 4 GB` 或 `4 vCPU / 8 GB` 起步
  - 数据层上托管 PostgreSQL
- 厂商优先级：
  1. `腾讯云`
  2. `阿里云`

### 为什么这样选

- 真实中国用户体验和合规路径会立刻变得重要。
- 一旦有真实数据，继续用本地文件存储就不合适了。
- 国内云更适合：
  - 中国内地网络体验
  - 域名解析与证书配套
  - 后续备案与生产升级

## 4.3 我的最终选型建议

- 如果你让我今天就拍板第一台机器：
  - 我建议用 `腾讯云 Lighthouse`
  - 规格选 `2 vCPU / 4 GB / 90 GB SSD / 30 Mbps`
- 原因：
  - 价格锚点清晰
  - 轻量应用场景匹配度高
  - 国内用户访问更友好
  - 后续升级到 `CVM + TencentDB + COS + CLB` 的路径自然

- 如果你明确说“我们先不走中国内地公开发布，只做内部试用”：
  - 我会把备选切成：
    - `腾讯云香港`
    - `DigitalOcean 新加坡`
    - `Hetzner`

- 如果你明确说“我们就是要走中国内地正式上线”：
  - 那我会建议直接按：
    - 国内节点
    - 托管数据库
    - ICP 准备
  来做，而不是继续把系统停留在 demo 架构。

## 5. 落地建议

- 下一步建议顺序：
  1. 先确认目标环境是 `内部 demo` 还是 `中国内地 pilot`
  2. 锁定厂商和地域
  3. 再做部署清单：
     - 操作系统
     - 反向代理
     - HTTPS
     - 环境变量
     - 进程管理
     - 数据备份
     - 数据库替换计划
